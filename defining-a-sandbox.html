Defining a Sandbox

Introduction

Defining a sandbox in the SP Framework is a critical step in creating a robust and maintainable control system. A sandbox establishes clear boundaries for your system, ensuring that it operates within specified parameters and meets the requirements of your project.

The sandbox definition process involves specifying parameters such as the minimum and maximum number of displays, input sources, supported classifications, and other system capabilities. By clearly defining these parameters, you create a controlled environment that ensures consistency and reliability across different installations.

Sandbox Configuration Basics

The core of a sandbox definition is the sandbox configuration dictionary, which is defined in the __sandbox_variables.py file. This file defines and manages variables used within the sandbox environment, including configuration settings and state information.

Sandbox Configuration Parameters

The sandbox configuration defines the "walls" or boundaries of your sandbox, establishing what the system can and cannot do. Common parameters include:

  • Display Limits - The maximum and minimum number of displays supported by the system
  • Input Limits - The maximum and minimum number of input sources supported
  • Classifications - The security classifications supported by the system
  • Room Information - Details about the room where the system is installed
  • Default Settings - Default values for various system parameters
  • Feature Flags - System capabilities and whether they are enabled

Sandbox Components

A sandbox in the SP Framework consists of several key components that work together to define the control system environment. These components are organized in a structured directory hierarchy to maintain clarity and separation of concerns.

Core Sandbox Files

The sandbox directory contains several core files that define the sandbox environment:

__sandbox_variables.py

This file defines and manages variables used within the sandbox environment, including configuration settings and state information. It contains the sandbox configuration dictionary that holds all parameters defining the boundaries and capabilities of your control system.

__sandbox_tools.py

This file provides a collection of tools specific to the sandbox environment, including testing, debugging, and development utilities. These tools help with sandbox creation, management, and troubleshooting.

__sandbox_moduleLibrary.py

This file contains the library of modules available within the sandbox environment, providing a controlled set of components for sandbox applications. It ensures that only approved and compatible modules are used in the control system.

Device Management

The sandbox includes a set of files for managing devices, located in the protected_device_files directory:

__sandbox_devices.py

This file defines the devices available within the sandbox environment, including their capabilities, limitations, and configuration options. It establishes what hardware can be used in the control system.

__sandbox_devices_csv.py

This file provides functions for importing and exporting device configurations using CSV files, enabling easy bulk configuration. This allows for device configurations to be managed outside the code.

__sandbox_module_components.py

This file defines the components that make up modules within the sandbox, including their interfaces, behaviors, and interactions. It establishes how modules work together within the sandbox environment.

Routing Management

The sandbox includes a file for managing routing configurations, located in the protected_routing_files directory:

__sandbox_routing_csv.py

This file provides functions for importing and exporting routing configurations using CSV files, enabling flexible signal routing setup. This allows for routing configurations to be managed outside the code.

For detailed information about device and matrix configurations, including file formats, examples, and usage, please refer to the Configuration Files documentation, which covers:

  • Device Configuration - How to define hardware devices in your system
  • Matrix Configuration - How to set up signal routing paths

Sandbox Implementation Workflow

Based on the structure of the sandbox files, implementing a sandbox follows this workflow:

1. Project Scope Definition

  • Define Room Type - Categorize spaces based on functionality and requirements
  • Determine Equipment Type - Identify the appropriate technology solutions for each space

2. Determine Equipment Footprint

  • Models of displays - Select appropriate display technologies and sizes
  • Cameras, Codecs - Choose video capture and processing equipment
  • Matrix Switchers - Determine signal routing and distribution needs
  • DSP's - Select audio processing solutions

3. Determine Sandbox Walls

  • Min-Max displays - Define the range of supported display configurations
  • Min-Max Inputs - Establish input capacity requirements and limitations
  • Supported Classifications - Identify security and compliance requirements

4. Determine Control Footprint / Deltas

  • Cameras - Define camera control requirements and presets
  • Codecs, Blu-ray - Configure media playback and conferencing controls
  • Audio, Time Zone Clocks - Set up audio management and time displays
  • Messaging - Implement notification and communication systems

Best Practices

When defining a sandbox, consider the following best practices:

  • Clear Configuration - Define clear parameters in your sandbox_config dictionary, including minimum and maximum values for key parameters.
  • Consistent CSV Formats - Maintain consistent formats for device and matrix configuration CSV files to ensure proper importing.
  • Organized Directory Structure - Follow the established directory structure with protected_device_files and protected_routing_files to maintain clarity.
  • Proper Module Registration - Ensure all modules are properly registered in the moduleLibrary for use within the sandbox.
  • Documentation - Document your sandbox configuration thoroughly, including the purpose of each file and configuration parameter.

Conclusion

Defining a sandbox is a critical step in creating a robust and maintainable control system with the SP Framework. The sandbox directory structure and files provide a structured approach to defining the boundaries and capabilities of your control system.

By properly configuring the sandbox files and following the established workflow, you can create control systems that are:

  • Scalable - Easily adapt to different room sizes and requirements by adjusting configuration files
  • Maintainable - Separate configuration from code for easier updates and modifications
  • Consistent - Ensure uniform behavior across different installations
  • Secure - Enforce security and compliance requirements through defined boundaries

The sandbox approach, with its structured files and clear separation of concerns, provides a solid foundation for building reliable and adaptable control systems.
