GUI Design Principles

Introduction

Effective GUI design is crucial for creating control systems that are both functional and user-friendly. The SP Framework provides a structured approach to GUI development that ensures consistency, scalability, and maintainability across different projects. This guide outlines the principles and best practices for designing GUIs that work seamlessly with the SP Framework.

Customization

Building touch panel GUIs that support your sandbox and all of its functionality requires a systematic approach. The SP Framework is designed to make this process more manageable by providing a structured way to create, customize, and maintain touch panel interfaces.

Understanding the Sandbox Environment

Before customizing your GUI, it's essential to understand how the sandbox environment works within the SP Framework:

  • The sandbox provides a controlled environment for testing and deploying control systems
  • It defines the boundaries of your system, including supported devices, inputs, outputs, and functionality
  • The sandbox configuration files determine what features are available to your GUI

GUI Development Process

When building a GUI for your sandbox, follow these steps:

  1. Plan Your Interface: Define the user requirements, workflow, and functionality needed
  2. Design Page Layout: Create wireframes for each page, considering user experience and workflow
  3. Implement Core Framework Pages: Ensure all required framework pages (prefixed with "__") are included
  4. Create Custom Pages: Develop your application-specific pages following the naming conventions
  5. Configure Element Properties: Set up all UI elements with proper IDs, names, and functionality
  6. Test Within Sandbox Limits: Verify that all GUI elements work within the defined sandbox boundaries

Leveraging Framework Features

The SP Framework provides several features that simplify GUI customization:

  • Global Functions: Access framework functions from any file to control UI elements
  • Dynamic Instantiation: Create UI elements programmatically based on configuration files
  • Centralized Element Management: Control UI elements from any part of your code
  • Consistent Styling: Apply uniform styling across all pages for a professional look

Integration with Sandbox Components

Your GUI needs to interact effectively with all components defined in your sandbox:

  • Connect UI elements to device controls defined in __sandbox_devices.py
  • Implement routing controls that work with the routing configuration in __sandbox_routing_csv.py
  • Utilize module components defined in __sandbox_module_components.py for specialized functionality
  • Leverage the tools provided in __sandbox_tools.py for common GUI operations

For more information on designing a sandbox visit What is a Sandbox?

Best Practices

Following these best practices ensures that your GUI will work reliably with the SP Framework and be easier to maintain over time:

Element Naming and ID Conventions

  • Blank Elements: If an element has no ID, use the name "blankSpace" to avoid conflicts. This ensures that every element has a unique identifier in the system.
  • Name and ID Consistency: The goal is to have no conflicts in names or element ID's. The name and ID must be the same for each element to ensure proper functionality within the framework.
  • Framework Pages: Any page with the prefix "__" (double underscore) is a framework page that cannot be removed. It is necessary for booting and ensuring the system loads correctly. These pages handle critical system functions and should never be modified or deleted.
  • ID Numbering Scheme: Every ID # uses the 1000 number scheme. Instead of numbering an element as 1, it should be numbered as 1001, and so on. This provides a consistent numbering system and helps avoid conflicts with system-reserved IDs.

Page and Panel Configuration

  • Default Page: Once the boot up is complete, the default page is always "00_Idle". This ensures a consistent starting point for all user interactions.
  • Boot Panel Alias: Boot up panel alias is always = "Panel_Main" to ensure system functionality during boot up. This is critical for the framework to properly initialize all components.
  • Auxiliary Panels: Aux panels (any panel other than the boot up panel) only need the following framework pages: "A__Logo" and "00_Idle". This minimizes resource usage while maintaining essential functionality.

Element Naming Scheme

The SP Framework system is based on names, meaning ID's and names must match. The prefix for all elements must be consistent with the documentation, with the following naming scheme:

prefix_category_differentiator_number

Common element prefixes include:

Element Type Prefix Example
Button btn btn_source_hdmi_1001
Slider sld sld_volume_main_1001

Implementation Examples

Here's an example of how to properly name and reference UI elements in your code: 

# Correct element naming in GUI design
btn_power_on_1001 = Button(name='btn_power_on_1001', id='btn_power_on_1001')
txt_status_power_1001 = Text(name='txt_status_power_1001', id='txt_status_power_1001')

# Accessing elements in code
def power_on_system():
    # Using the element's full name
    element = tlp.GetElement('btn_power_on_1001')
    element.SetState(1)

    # Update status text
    status_text = tlp.GetElement('txt_status_power_1001')
    status_text.SetText('System Powering On...')

Common Pitfalls to Avoid

  • Using inconsistent naming patterns across different pages
  • Creating elements without following the proper ID numbering scheme
  • Removing or modifying framework pages (those with "__" prefix)
  • Using duplicate IDs or names across different elements
  • Forgetting to set the boot panel alias to "Panel_Main"
  • Creating custom default pages instead of using "00_Idle"

Conclusion

Following these GUI design principles and best practices will help you create touch panel interfaces that work seamlessly with the SP Framework. By maintaining consistent naming conventions, proper element configuration, and adherence to framework requirements, you'll develop GUIs that are robust, maintainable, and provide an excellent user experience.

Remember that the SP Framework is designed to make GUI development more structured and less error-prone. By leveraging its features and following these guidelines, you can focus more on creating intuitive user experiences and less on troubleshooting technical issues.
